# Qwen3-1.7B 端云协同 + Sparse Checkpoint 技术报告

## 实验模型

- **Qwen3-1.7B-MLX-4bit**: 28 layers, d=2048, 8 KV heads, head_dim=128
- GQA 2:1 (n_q=16, n_kv=8)
- 模型权重: ~872 MB (Q4 全模型), ~458 MB (半模型 layers 14-27)
- KV 每 token 每层: 4,096 B (2 × 8 × 128 × 2B, bf16)
- h^(0) 每 token: 4,096 B (2048 × 2B)
- tie_word_embeddings: true (无独立 lm_head, 用 embed_tokens.as_linear)
- h^(0) 压缩比: **28×** (Tier A)
- 实验平台: Apple Silicon Mac mini 48GB

---

## 1. Sparse Checkpoint 基准测试 (单设备)

### 1.1 配置矩阵

| 配置 | Checkpoint 层 | 传输量 (2K) | 压缩比 | 重建耗时 (2K) | 组大小 |
|------|--------------|-------------|--------|-------------|--------|
| [0] | h^(0) only | 8 MB | 28× | 935 ms | 28L |
| [0,14] | h^(0) + h^(14) | 16 MB | 14× | 1,025 ms | 14L |
| [0,7,14,21] | 4 checkpoints | 32 MB | 7× | 989 ms | 7L |
| [0,4,8,12,16,20,24] | 7 checkpoints | 56 MB | 4× | 1,033 ms | 4L |

### 1.2 不同 Context 长度

| 配置 | 512 | 1024 | 2048 | 质量 |
|------|-----|------|------|------|
| [0] | 237 ms | 473 ms | 935 ms | 28/28 EXACT |
| [0,14] | 263 ms | 517 ms | 1,025 ms | 28/28 EXACT |
| [0,7,14,21] | 252 ms | 501 ms | 989 ms | 28/28 EXACT |
| [0,4,8,12,16,20,24] | 261 ms | 527 ms | 1,033 ms | 28/28 EXACT |

### 1.3 单设备结论

**所有配置总重建时间几乎相同** (~935-1033 ms at 2K)。

原因: 单设备上总 forward 层数不变 (都是 28L)。多组有轻微开销 (~10%)，来自:
- mx.eval 同步次数增加
- 多组 temp cache 分配/释放
- 组间切换开销

**单设备推荐**: `[0]` (h^(0) only) — 最简单，最高压缩比 (28×)，速度最快。

## 2. 端云协同方案对比

### 2.1 五种方案

| 方案 | 描述 | 云端发送 | 传输量 (4K) | 端侧重建 | 端侧模型 |
|------|------|----------|-------------|---------|----------|
| **A** | 全量本地 | — | — | 28L prefill | 872 MB |
| **B** | Pipeline@14 原版 | h^(14) + KV[0:13] int8 | 135 MB | 14L | 458 MB |
| **C** | h^(0) only | h^(0) | 16 MB | 28L | 872 MB |
| **E** | Full KV (DistServe) | 全量 KV int8 | 252 MB | 0L | 872 MB |
| **G** | [0,14] + Pipeline@14 | h^(0) + h^(14) | 32 MB | 28L (2×14L) | 872 MB |

### 2.2 实测数据 (4K context)

| 方案 | 传输量 | 重建耗时 | 质量 |
|------|--------|---------|------|
| A) 全量本地 | — | 2,847 ms (全量 prefill) | baseline |
| B) Pipeline@14 | 135 MB (int8) | 1,554 ms (14L) | 20/20 EXACT |
| C) h^(0) only | 16 MB | 2,847 ms (28L) | 20/20 EXACT |
| E) Full KV | 252 MB (int8) | 0 ms | 20/20 EXACT |
| G) [0,14]+Pipe@14 | 32 MB | 2,221 ms (2×14L) | 20/20 EXACT |

> G 重建比 C 快 22% (2,221 vs 2,847 ms)。两组 14L 内存压力更小，cache 效率更高。

### 2.3 TTFT 网络分析

**公式**: `TTFT = cloud_overhead (100ms) + transfer / bandwidth + recon_time`

| 网络 | 带宽 | B) Pipe@14 | G) [0,14]+Pipe@14 | C) h^(0) only | E) Full KV | 赢家 |
|------|------|-----------|-------------------|---------------|-----------|------|
| **LAN 1Gbps** | 125 MB/s | **2.73s** | **2.58s** | 3.08s | 2.12s | E > G > B > C |
| **WiFi6 200Mbps** | 25 MB/s | 7.05s | **3.60s** | **3.59s** | 12.18s | C ≈ G |
| **5G 100Mbps** | 12.5 MB/s | 12.45s | 4.88s | **4.23s** | 22.26s | **C** |
| **4G 20Mbps** | 2.5 MB/s | 55.65s | 15.12s | **9.35s** | 102.9s | **C** |
| **3G 4Mbps** | 0.5 MB/s | 271.6s | 66.3s | **34.9s** | 506.1s | **C** |

### 2.4 带宽交叉点分析

```
G vs C 交叉点:
  G 多传输: 16 MB, 省重建: 626 ms
  交叉点 = 16 MB / 0.626s = 25.6 MB/s ≈ 200 Mbps
  → 带宽 > 200 Mbps: G 略优
  → 带宽 < 200 Mbps: C 胜出

B vs G 交叉点:
  B 多传输: 103 MB, 省重建: 667 ms
  交叉点 = 103 MB / 0.667s = 154 MB/s ≈ 1.2 Gbps
  → 带宽 > 1.2 Gbps: B 略优 (但差异 <0.2s)
  → 带宽 < 1.2 Gbps: G 远优于 B

E vs C 交叉点:
  E 多传输: 236 MB, 省重建: 2,847 ms
  交叉点 = 236 MB / 2.847s = 82.9 MB/s ≈ 660 Mbps
  → 带宽 > 660 Mbps: E 最优 (零重建)
  → 带宽 < 660 Mbps: C 胜出
```

## 4. 方案决策树

```
端侧内存紧张? (装不下全模型)
  ├─ YES → B (Pipeline@14 原版)
  │         半模型 458 MB, 传输 135 MB int8
  │         适合: 小内存设备 (4GB RAM 手机)
  │
  └─ NO → 网络带宽?
           ├─ > 660 Mbps (LAN/光纤)
           │     → E (Full KV DistServe)
           │       零重建, TTFT 最低 (~2.1s)
           │
           ├─ 200-660 Mbps (快速 WiFi)
           │     → G ≈ C (几乎打平)
           │       G 快 ~0.1s 但多传 16 MB
           │       推荐 C (更简单)
           │
           └─ < 200 Mbps (5G/4G/3G)
                 → C (h^(0) only)
                   传输最少 (16 MB), TTFT 最优
                   5G: 4.2s, 4G: 9.4s
```

## 5. 与 Qwen3-8B 对比

| 指标 | Qwen3-1.7B | Qwen3-8B |
|------|-----------|----------|
| 层数 | 28 | 36 |
| d_model | 2048 | 4096 |
| GQA 比 | 2:1 | 4:1 |
| h^(0) 压缩比 | 28× (Tier A) | 18× (Tier B) |
| h^(0) per token | 4 KB | 8 KB |
| KV per token/layer | 4 KB | 4 KB |
| h^(0) @ 4K | 16 MB | 32 MB |
| Full KV @ 4K | 448 MB | 576 MB |

> Qwen3-1.7B 的 GQA 2:1 使其 h^(0) 压缩比更高 (28× vs 18×)。
> h^(0) 只有 d_model 大小，不受 n_kv_heads 影响 — GQA 越激进，h^(0) 越有优势。

## 6. 核心结论

### 6.1 h^(0) only (C) 是端云协同最优解

在绝大多数真实端云场景 (5G/4G/WiFi) 下:
- **传输最少**: 16 MB (仅 embed_tokens 输出)
- **质量完美**: EXACT, 零 token 偏差
- **实现最简**: 无需 KV 量化/传输/注入，云端只需跑 embed_tokens

### 6.2 [0,14]+Pipeline@14 (G) 是有价值的优化，但场景有限

G 比 C 快 22% 重建 (两组14L vs 一组28L)，但:
- 多传 16 MB → 低带宽场景反而更慢
- 只在 > 200 Mbps 时有 ~0.5s 优势
- 此时 TTFT 已经 < 3.6s，0.5s 差异体感不大

### 6.3 Pipeline@14 (B) 的独特价值是**内存**

B 是唯一支持半模型的方案 (458 MB vs 872 MB)。
对内存受限设备 (4GB RAM 手机), B 可能是唯一可行方案。
TTFT 代价大 (5G: 12.5s vs C 的 4.2s)，但至少能跑。

### 6.4 Sparse Checkpoint 在单设备无收益

单设备上 [0], [0,14], [0,7,14,21] 总重建时间几乎相同。
Sparse checkpoint 的价值在于:
- 端云分组传输 (云端可以先发 h^(0), 端侧提前开始重建)
- 多设备并行重建 (数据中心 PD 分离)
- 不在于单设备加速
