# FlashMLX 铁律：Profiler First

**建立日期**: 2026-03-18
**优先级**: 🔥 最高
**违反后果**: 浪费时间在无效优化上

---

## 📜 铁律声明

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│                    PROFILER FIRST LAW                           │
│                                                                 │
│   任何优化前必须先用 Profiler 分析                              │
│                                                                 │
│   1. 优化前 → 先 Profile → 找瓶颈 → 再优化                     │
│   2. 优化后 → 再 Profile → 验证效果                            │
│   3. 分析不到位 → 先优化 Profiler → 再优化系统                 │
│                                                                 │
│   证据驱动 > 直觉驱动                                           │
│   数据优先 > 经验优先                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🎯 核心原则

### 原则 1: 证据驱动优化 (Evidence-Driven Optimization)

**定义**: 所有优化决策必须基于 Profiler 数据

**执行标准**:
- ✅ 有 profiling 数据 → 可以优化
- ❌ 没有数据 → 禁止优化

**示例**:
```
❌ 错误流程:
"我觉得 Flash Attention 可能慢"
→ 直接修改 kernel
→ 发现没效果
→ 浪费时间

✅ 正确流程:
Profile baseline
→ 发现 Flash Attention 占 45% 时间
→ 分析 kernel 瓶颈
→ 针对性优化
→ Profile 验证提升 15%
```

### 原则 2: 瓶颈优先 (Bottleneck First)

**定义**: 优先优化占比最高的操作

**执行标准**:
- ✅ 占比 > 20% → 高优先级
- ⚠️  占比 5-20% → 中优先级
- ❌ 占比 < 5% → 不值得优化

**80/20 法则**:
- 80% 的时间花在 20% 的代码上
- 优化这 20% 的代码能获得最大收益

**示例**:
```
Profiling 结果:
- flash_attention: 45% → 🔥 立即优化
- matmul: 30% → 🔥 立即优化
- gemv: 12% → ⚠️  考虑优化
- rms_norm: 8% → ⚠️  考虑优化
- rope: 3% → ❌ 暂不优化
- others: 2% → ❌ 暂不优化
```

### 原则 3: Profiler 迭代优化 (Profiler-First Iteration)

**定义**: Profiler 不够精确时，优先优化 Profiler

**执行标准**:
```
if Profiler 分析不到位:
    停止系统优化
    优化 Profiler (增加插桩点、提高精度)
    重新 Profile
else:
    基于数据优化系统
```

**触发条件**:
- Profiler 无法定位瓶颈
- 数据粒度太粗
- 缺少关键指标 (GPU 时间、内存带宽等)

**示例**:
```
场景: Profile 显示 "matmul 占 30%"，但不知道瓶颈在哪

❌ 错误做法:
盲目尝试各种 matmul 优化 (GEMM 算法、tile size、etc.)

✅ 正确做法:
1. 优化 Profiler: 增加 Metal kernel 级别追踪
2. 重新 Profile: 发现 gemv_4bit kernel 占 matmul 的 80%
3. 针对性优化 gemv_4bit
4. 验证提升
```

## 🚫 禁止模式

### 禁止 1: 凭感觉优化

```
❌ "我觉得这里可能慢，试试优化"
❌ "这个算法理论上更快"
❌ "别人项目这么做的"
```

**后果**: 浪费时间在不是瓶颈的地方

### 禁止 2: 没有 Baseline 对比

```
❌ 优化后: "快了！"（没有数据）
❌ 优化后: "应该快了吧？"（不确定）
```

**后果**: 无法验证效果，可能引入回退

### 禁止 3: 优化占比 < 5% 的操作

```
❌ rope 占 3% → 花 2 天优化 → 提升 50% → 端到端只提升 1.5%
```

**后果**: ROI 太低，得不偿失

### 禁止 4: 盲目照搬其他项目

```
❌ "PyTorch 用了 Flash Attention 2，我们也用"
   （不知道 MLX 是否已经用了）

❌ "ThunderLLAMA 用了 K/V Fusion，我们也用"
   （不知道 MLX 架构是否适合）
```

**后果**: 可能不适用，甚至负优化

## ✅ 正确流程

### 标准优化流程

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: Profile Baseline                                    │
│ with Profiler("baseline"):                                  │
│     output = model.generate(prompt)                         │
│ → 生成 baseline.json                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 分析瓶颈                                            │
│ report = analyze("baseline.json")                           │
│ → Top 3 瓶颈: flash_attention (45%), matmul (30%), ...     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 判断 Profiler 是否足够精确                          │
│ if 可以定位具体瓶颈:                                        │
│     继续 Step 4                                             │
│ else:                                                       │
│     优化 Profiler → 回到 Step 1                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 设计针对性优化                                      │
│ 目标: flash_attention (45%)                                 │
│ 方案: 优化 Metal kernel tile size                           │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 实现优化                                            │
│ 修改 scaled_dot_product_attention.metal                     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 6: Profile Optimized                                  │
│ with Profiler("optimized"):                                 │
│     output = model.generate(prompt)                         │
│ → 生成 optimized.json                                       │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 7: 对比验证                                            │
│ compare_profiles("baseline.json", "optimized.json")         │
│ → flash_attention: 45ms → 38ms (-15.5%)                    │
│ → 总时间: 100ms → 85ms (-15%)                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 8: 决策                                                │
│ if 提升 > 3%:                                               │
│     提交 PR (附带对比数据)                                  │
│ else:                                                       │
│     放弃优化 (ROI 太低)                                     │
└─────────────────────────────────────────────────────────────┘
```

## 📊 PR 提交要求

### 优化 PR 必须包含

1. **Baseline Profiling 数据**
   - `baseline_profile.json`
   - 瓶颈分析 (Top 3 热点)

2. **优化说明**
   - 优化目标 (哪个函数/kernel)
   - 优化方案 (具体改动)
   - 理论分析 (为什么会快)

3. **Optimized Profiling 数据**
   - `optimized_profile.json`
   - 对比分析 (before/after)

4. **验收标准**
   - 端到端提升 > 3%
   - 目标函数提升 > 10%
   - 无性能回退 (其他函数)

### PR 模板

```markdown
## 优化目标
- 函数: flash_attention
- Baseline 占比: 45% (45ms)
- 目标提升: > 10%

## 优化方案
- 修改 Metal kernel tile size: 16x16 → 32x32
- 减少内存带宽消耗

## Profiling 数据
### Baseline
- Total: 100ms
- flash_attention: 45ms (45%)

### Optimized
- Total: 85ms (-15%)
- flash_attention: 38ms (-15.5%)

### 对比
- ✅ flash_attention: -15.5%
- ✅ 端到端: -15%
- ✅ 无其他回退

## 附件
- baseline_profile.json
- optimized_profile.json
- compare_report.md
```

## 🎓 教训来源

### ThunderLLAMA 失败案例

**问题**: 尝试了 10+ 个优化，只有 2 个有效

**原因**:
1. 没有精确的 profiling 数据
2. 凭感觉优化 RMSNorm、SWIGLU 等小操作
3. 浪费时间在占比 < 3% 的函数上

**结果**:
- MoE Fusion: -0.17% (负优化！)
- SWIGLU Vectorization: +0.06% (几乎没用)
- Add+RMSNorm Fusion: +0.21% (微不足道)

**教训**:
- 如果有 Profiler，会立即发现这些操作占比太低
- 不会浪费时间优化它们
- 会直接聚焦 MoE FFN MatMul (50-60%)

### 正确案例

**K/V Projection Fusion**:
1. Profile: 发现 K/V Projection 占 12%
2. 优化: 融合 QKV 三个 projection
3. 验证: +9.8% TG, +8.5% PP
4. 结论: 值得投入

## 🔍 自检清单

优化前自检：
- [ ] 我跑了 Profiler 吗？
- [ ] 我知道瓶颈在哪吗？
- [ ] 瓶颈占比 > 5% 吗？
- [ ] Profiler 够精确吗？
- [ ] 如果不够精确，我优化 Profiler 了吗？

优化后自检：
- [ ] 我跑了 Profiler 验证吗？
- [ ] 有 before/after 对比数据吗？
- [ ] 提升 > 3% 吗？
- [ ] 有其他回退吗？
- [ ] PR 附带了 profiling 数据吗？

---

## 📝 违反后果

| 违反行为 | 后果 |
|----------|------|
| 没有 Profile 就优化 | 浪费时间，可能无效 |
| 没有对比数据 | 无法验证效果 |
| 优化占比 < 5% 的操作 | ROI 太低 |
| Profiler 不够精确不优化 | 盲目优化，效率低 |
| 凭感觉/经验优化 | 可能南辕北辙 |
