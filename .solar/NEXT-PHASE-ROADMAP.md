---

## 视角转变：代码补丁 vs 架构级

| 类别 | 之前的分析 | 现在的分析 |
|------|-----------|-----------|
| **视角** | 函数级 | 系统级 |
| **方法** | "这个循环可以优化" | "800 次内核启动才是瓶颈" |
| **量化** | "预估 5-15%" | "62% 带宽利用率，理论极限 150 tok/s" |
| **影响** | 代码行数 | 架构范式 |

---

## 架构级优化 (A1-A5)

| ID | 优化 | 收益预估 | 说明 |
|----|------|---------|------|
| **A1** | Layer Fusion | +12-18% TG | 从根源解决内核调度开销 |
| **A2** | 连续批处理 | 8× 吞吐 | 改变整个 serving 经济模型 |
| **A3** | 计算流水线 | 62%→85% 利用率 | 逼近硬件天花板 |
| **A4** | 动态剪枝 | 150→175 tok/s | 减少不必要的计算 |
| **A5** | Expert Streaming | 释放 9.7 GB | UMA 架构的独特优势 |

### 实施路径

```
Phase 1 (立即可做): A1 Layer Fusion + A4 动态剪枝
  → 2-3 天，单请求 TG: 93→120+ tok/s

Phase 2 (中期): A3 计算流水线
  → 1 周，单请求 TG: 120→135 tok/s

Phase 3 (架构升级): A2 连续批处理 + A5 Streaming
  → 2 周，总吞吐: 135→600+ tok/s (多并发)
```
