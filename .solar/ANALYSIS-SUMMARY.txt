================================================================================
问题 1：为什么 AM 在真实数据上质量从 0.9999 降至 0.898？
================================================================================

根本原因：Beta 补偿的自由度不足

数学分析：
  - 约束数量：n (query samples) × t (compressed size) = 20 × 30 = 600
  - 自由度：t (beta parameters) = 30
  - 比值：600/30 = 20（严重欠定）
  
  → β 有 30 个参数，需要补偿 600 个约束（不可能）

真实数据特性：
  - 注意力集中（entropy=3.42 vs 模拟数据 4.61）
  - 关键 token 占 80%+ 权重
  - 其他 token 权重接近 0
  
  → 移除一个关键 token → 损失 80% 信息
  → β 最多补偿 10-20% → 无法弥补

为什么 H2O/StreamingLLM 更好（0.94-0.95）：
  - H2O：明确保留 heavy-hitter token（正好是关键的）
  - StreamingLLM：保留 sink + recent（经验上有效）
  - 都不依赖复杂的 β 补偿机制

================================================================================
问题 2：为什么 TruthfulQA 完全失败（质量=0.000）？
================================================================================

直接原因：t > T 的逻辑矛盾

T = 15 (序列长度)
t = max(25, 15//4) = max(25, 3) = 25
结果：无法从 15 个 token 选出 25 个！

执行流程：
  1. topk(25, on 15 tokens) → 只返回 15 个索引
  2. NNLS 求解 25 个参数的方程，但矩阵秩只有 15
  3. NNLS 返回 B = zeros 或 NaN（rank-deficient solution）
  4. β = log(B) = -∞ 或 NaN
  5. softmax(... + (-∞)) = 0 → 质量 = 0

设计缺陷：
  - 压缩目标 t 计算无视序列长度的下限
  - 应该是：t = min(target, T) 而不是 max(25, T//4)
  - 但即使修复，短序列的集中注意力问题仍存在

================================================================================
关键洞察：AM 的三个核心假设全部违反
================================================================================

假设                | 模拟数据 ✅  | 真实数据 ❌  | 原因
================================================================================
注意力分散          | Yes         | No          | 真实是峰值分布
β 可补偿缺失        | 容易        | 很难        | 移除关键 token
t < T（可压缩）     | Yes         | No (短序列) | T=15 时 t>T

结论：AM 的设计针对"不存在的场景"（均匀注意力），而真实 LLM 场景完全不同

为什么 H2O/StreamingLLM 更稳健？
================================================================================

1. 不依赖复杂的数学假设，而是经验驱动的启发式
2. Heavy-Hitter 概念与真实注意力峰值对齐
3. 直接保留原始 token，不试图通过参数补偿
4. 无数值精度问题（不涉及 exp/log 的条件数灾难）
5. 在所有注意力分布上表现一致

================================================================================
Qwen3.5 混合架构为什么完全崩溃？
================================================================================

执行链：
  Layer 3 (Attention, 压缩)  
    → 压缩误差 ε ≈ 0.05-0.1
  Layer 4-6 (SSM, 无压缩)
    → SSM 递推接收污染的输入
    → 误差被放大或累积
  Layer 7 (Attention, 压缩)
    → 新的压缩误差叠加
  ...repeat...
  
  最终：累积误差呈指数增长 → 生成完全乱码

根本问题：Attention 压缩的微小误差在 SSM 递推中被放大

================================================================================
数值精度的角色（次要但存在）
================================================================================

Float16 问题：
  - exp(logits) 范围 [1e-22, 1e22]，float16 无法表示
  - 导致溢出（Inf）或下溢（0）
  - NNLS 矩阵条件数 > 1e10，float32 精度不足

但这是次要原因，因为：
  - 代码已转为 float32（见行 329-331）
  - 问题仍然存在，说明核心是算法，不是精度

关键教训
================================================================================

1. 模拟数据的陷阱
   - 均匀注意力是特例，不是规律
   - 不要被模拟完美表现迷惑

2. 算法假设必须验证
   - 在真实数据上检验每个关键假设
   - 假设失败 → 算法失败，非局部问题

3. 启发式 > 优化（在不确定性高时）
   - H2O/StreamingLLM 无假设，工程鲁棒
   - AM 有假设，假设违反则完全失败

4. 跨层交互很重要
   - 混合架构中，层与层的误差交互非平凡
   - 单层优化不保证端到端质量
